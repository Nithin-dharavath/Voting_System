     Implementation Plan: Candidate Application

     Context

     This feature allows students to apply as candidates for upcoming elections.
     Students provide a manifesto, and their application remains in a PENDING
     state until an administrator reviews it. Only APPROVED candidates can
     receive votes. This implementation adds the necessary routes, business
     logic, and UI templates to support the application flow.

     Implementation Approach

     1. Backend Implementation (app.py)

     A. Route: GET /student/elections/{id}/apply

     - Access Control: Apply student_guard.
     - Business Logic:
       - Fetch election by id. Redirect to /student/elections if not found.
       - Validate that election.status == 'UPCOMING'. If not, redirect to
     /student/elections.
       - Check if the student has already applied to this election using a query
     on candidate_applications. If yes, redirect to /student/candidate-status.
     - UI: Generate a CSRF token and render
     templates/student_election_apply.html.

     B. Route: POST /candidates/apply

     - Access Control: Apply student_guard and verify_csrf.
     - Business Logic:
       - Validate election_id and manifesto from the form request.
       - Re-validate that the election exists and is UPCOMING.
       - Re-validate that the student has not already applied (to prevent
     duplicate/direct POST submissions).
       - Insert a new record into candidate_applications with user_id,
     election_id, manifesto, and approval_status = 'PENDING'.
     - Outcome: RedirectResponse to /student/candidate-status.

     C. Route: GET /student/candidate-status

     - Access Control: Apply student_guard.
     - Business Logic:
       - Query candidate_applications joined with elections to retrieve all
     applications for the logged-in user.
       - Normalize dates using ensure_datetime().
     - UI: Render templates/student_candidate_status.html with the list of
     applications.

     D. Modify GET /student/elections/{id}

     - Update Logic:
       - Add a query to check if the currently logged-in student has applied for
     the requested election.
       - Pass a has_applied boolean to the template context.

     2. Template Implementation

     A. templates/student_election_apply.html (New)

     - Extends base.html.
     - Form containing:
       - Election title.
       - Textarea for manifesto.
       - Hidden inputs for election_id and csrf_token.
       - Submit button.

     B. templates/student_candidate_status.html (New)

     - Extends base.html.
     - Table displaying the student's candidate applications:
       - Election Title.
       - Application Status (with semantic colors for PENDING, APPROVED,
     REJECTED).
       - Application Date.
     - Empty state message for users with no applications.

     C. templates/student_election_detail.html (Modify)

     - Update the "Apply as Candidate" button logic:
       - Only show if election.status == 'UPCOMING' and not has_applied.

     Critical Files

     - app.py
     - templates/student_election_apply.html
     - templates/student_candidate_status.html
     - templates/student_election_detail.html

     Verification Plan

     1. Happy Path:
       - Navigate to an UPCOMING election detail page $\rightarrow$ Click "Apply
     as Candidate" $\rightarrow$ Submit manifesto $\rightarrow$ Verify redirect
     to status page and record insertion in DB.
     2. Eligibility Checks:
       - Attempt to access the apply form via URL for an ACTIVE or ENDED
     election $\rightarrow$ Verify redirect.
       - After applying, verify "Apply as Candidate" button is hidden on the
     detail page.
       - Attempt to re-submit the apply form for the same election $\rightarrow$
     Verify redirect to status page.
     3. Security:
       - Test POST /candidates/apply without a CSRF token $\rightarrow$ Verify
     failure.
       - Test POST /candidates/apply with a non-student account $\rightarrow$
     Verify redirect.
     4. UI:
       - Verify the status page correctly displays election names and status
     labels.

‚óŹ The error you are seeing is occurring because you are visiting
  http://127.0.0.1:8000/student/elections/apply, but the application is designed
  so that you apply for a specific election.

  The route is defined as /student/elections/{id}/apply. Because the URL you
  entered matches the pattern of the election detail page
  (/student/elections/{id}), FastAPI is trying to treat the word "apply" as the
  election id. Since the id is expected to be an integer, the "int_parsing"
  error is thrown.

  How to fix this:

  You must include the ID of the election you want to apply for in the URL. For
  example, if the election ID is 1, the correct URL is:
  http://127.0.0.1:8000/student/elections/1/apply

  To test this correctly:
  1. Go to the Student Elections list: http://127.0.0.1:8000/student/elections
  2. Click on a specific Upcoming election.
  3. Click the "Apply as Candidate" button on that election's detail page. This
  will take you to the correct URL automatically.
